## Root Cause

The vulnerability is a double free in `dialog_changed` triggered when:
1. A buffer with empty filename (`b_fname == NULL`) is saved
2. `buf_set_name` (line 649) sets `buf->b_fname = buf->b_sfname` after calling `fname_expand`, causing both fields to point to the **same allocation** (line 513-514 in `fname_expand` sets `*sfname = *ffname` when sfname is NULL)
3. When the write operation fails, `VIM_CLEAR(buf->b_fname)` at line 661 frees the shared memory
4. Then `VIM_CLEAR(buf->b_sfname)` at line 663 attempts to free the **same memory again**, causing a double free

The `VIM_CLEAR` macro at lines 202-209 calls `vim_free()` and sets the pointer to NULL, but since `b_fname` and `b_sfname` point to the same memory after `fname_expand`, the second free targets memory already freed by the first call.

## Trigger Path

1. `bad.c:620` `dialog_changed`
   - Entry point function that prompts user to save changes to a buffer
   - Checks if `b_fname == NULL` (empty buffer), saves this condition at line 647

2. `bad.c:649` `buf_set_name(buf->b_fnum, (char_u *)"Untitled")`
   - Assigns an "Untitled" name to buffer; internally sets `b_fname = b_sfname` after calling `fname_expand` so both point to same allocation

3. `bad.c:505-516` `fname_expand(buf, &buf->b_ffname, &buf->b_sfname)`
   - At lines 513-514: `if (*sfname == NULL) *sfname = *ffname;` causes `b_sfname` and `b_ffname` to share the same pointer

4. `bad.c:654` `buf_write_all(buf, FALSE)`
   - Attempts to write the buffer; may fail for various reasons

5. `bad.c:661` `VIM_CLEAR(buf->b_fname)` (sink #1)
   - First free: frees shared allocation that `b_fname` points to

6. `bad.c:663` `VIM_CLEAR(buf->b_sfname)` (sink #2 / vulnerable double free)
   - Second free: attempts to free what is now a dangling pointer to the same memory already freed at line 661
