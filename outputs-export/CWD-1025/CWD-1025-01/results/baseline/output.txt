

## Root Cause

The vulnerability is a double free in the `CacheEntry` object stored in the `serviceCache` Hashtable. At line 830, `serviceCache->put(result->actualDescriptor, result, status)` stores the CacheEntry pointer in the hashtable with `cacheDeleter` set as the value deleter (line 754). The `cacheDeleter` calls `CacheEntry::unref()` which deletes the object when the reference count drops to zero. If the subsequent `U_FAILURE(status)` check at line 831 detects failure, `delete result` at line 832 frees the CacheEntry directly. Later, when the `serviceCache` Hashtable is destroyed, the value deleter (`cacheDeleter`) attempts to delete the same `result` object again, causing a double free. The source `(hash->valueDeleter)(valuepointer)` at lines 157-158 in the `HASH_DELETE_KEY_VALUE` macro is the location where the second free occurs during hashtable cleanup.

## Trigger Path

1. `serv.cpp:729` `ICUService::getKey(ICUServiceKey& key, UnicodeString* actualReturn, const ICUServiceFactory* factory, UErrorCode& status)`
   - External entry point that receives the key and status; calls internal logic to get or create a CacheEntry result.

2. `serv.cpp:830` `serviceCache->put(result->actualDescriptor, result, status)`
   - Stores the CacheEntry pointer in the Hashtable with cacheDeleter set as the value deleter; control flows based on the success/failure of this operation.

3. `serv.cpp:831-832` `if (U_FAILURE(status)) { delete result; }`
   - Sink: If status indicates failure (e.g., from memory allocation error), deletes result directly, freeing the CacheEntry.

4. `uhash.cpp:157-158` `HASH_DELETE_KEY_VALUE` macro `(hash->valueDeleter)(valuepointer);`
   - Later during Hashtable destruction, this source location calls the valueDeleter to free stored values, causing the second free of the already-deleted CacheEntry.
