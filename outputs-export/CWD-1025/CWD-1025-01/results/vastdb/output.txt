## Root Cause

The vulnerability is in `ICUService::getKey` (bad.cpp:729). A `CacheEntry* result` object is stored in the hashtable via `serviceCache->put()` at line 830. If the put operation fails (U_FAILURE at line 831), `delete result` is called at line 832 to free the object. However, before returning NULL on error, `_uhash_put` already calls `HASH_DELETE_KEY_VALUE` at line 722 which invokes the valueDeleter (`cacheDeleter`) on the value if the operation fails. The `cacheDeleter` calls `CacheEntry::unref()` which decrements the refcount and calls delete when it reaches zero. This creates a double free: when `unref()` deletes the object and then the explicit `delete result` at line 832 tries to free it again.

## Trigger Path

1. `bad.cpp:729` `icu::ICUService::getKey`
   - Entry function that processes service cache lookups and stores results in the hashtable.

2. `bad.cpp:830` `serviceCache->put(result->actualDescriptor, result, status)`
   - Stores the CacheEntry* result in the serviceCache hashtable. The put internally calls `_uhash_put`.

3. `bad.cpp:831` `if (U_FAILURE(status))`
   - Checks if the put operation failed. If error occurred, goto error path.

4. `bad.cpp:722` `_uhash_put` error handler
   - When `_uhash_put` fails, HASH_DELETE_KEY_VALUE macro invokes `hash->valueDeleter(valuepointer)` which is `cacheDeleter` (line 589).

5. `bad.cpp:589` `cacheDeleter` / `bad.cpp:573` `CacheEntry::unref()`
   - Calls unref() which deletes the CacheEntry when refcount reaches zero (line 575: `delete this;`).

6. `bad.cpp:832` `delete result;`
   - After the put failure, the explicit delete tries to free the already-deleted result object, causing the double free.

The trigger path shows the value gets freed first by `unref()` in the error handler path, then the explicit `delete result` tries to free it again.
