## Root Cause

The vulnerability is a double-free triggered by the interaction between `extract_lltask_from_task` and its caller `dnn_execute_model_tf`. When `ff_queue_push_back(lltask_queue, lltask)` succeeds at line 229 in `extract_lltask_from_task`, the `lltask` object (which stores a reference to the `task` pointer via `lltask->task = task` at line 228) is successfully added to the queue. However, if the function later returns an error (e.g., from a subsequent operation), `av_freep(&task)` is called at line 271 in the caller, freeing `task` while `lltask` still holds a reference to it. When the queue is later processed and `lltask` is freed, it attempts to free the already-freed `task` pointer again, causing a double-free. Additionally, if `ff_queue_push_back` fails inside `extract_lltask_from_task`, `lltask` is freed at line 231, but the caller may also attempt to free `task` at lines 259, 264, 271, or 278 depending on control flow, creating another double-free path.

## Trigger Path

1. `bad.c:237` `dnn_execute_model_tf`
   - Allocates `task` via `av_malloc` at line 250
   - Calls `extract_lltask_from_task(task, ...)` at line 269

2. `bad.c:217` `extract_lltask_from_task`
   - Allocates `lltask` via `av_malloc` at line 221
   - Stores reference: `lltask->task = task` at line 228 (task pointer saved in lltask)
   - Pushes `lltask` to queue at line 229 via `ff_queue_push_back`

3. Trigger paths:
   - **Source path (line 231):** If `ff_queue_push_back` fails, `av_freep(&lltask)` frees lltask at line 231. The caller may later free `task` at line 271 if `extract_lltask_from_task` returns error.
   - **Sink path (line 271):** Caller frees `task` via `av_freep(&task)` when `extract_lltask_from_task` returns error, but `lltask` in the queue still holds a reference to `task` if push was successful, causing double-free when queue is later processed.
