

## Root Cause

The vulnerability is a double free (CWE-415) where `task` is freed twice. At line 263, `task` is pushed to `tf_model->task_queue` via `ff_queue_push_back`. However, immediately after at line 264, `av_freep(&task)` frees the memory - but the queue still holds a reference to this now-freed pointer. Later, when `extract_lltask_from_task` returns with an error (due to the internal `av_freep(&lltask)` at line 231), line 271 calls `av_freep(&task)` again on the already-freed memory, triggering the double free.

## Trigger Path

1. `bad.c:237` `dnn_execute_model_tf`
   - Allocates `task` at line 250, pushes to queue at line 263, then frees at line 264 before calling `extract_lltask_from_task`

2. `bad.c:217` `extract_lltask_from_task`
   - Receives `task` as parameter, allocates and queues `lltask` at lines 221-229, then frees `lltask` at line 231 due to queue push failure, returns error - control returns to caller with `task` still referenced in queue but freed

3. `bad.c:271` `av_freep(&task)` (sink)
   - Frees `task` again after `extract_lltask_from_task` returns error, causing double free since `task` was already freed at line 264
