


Now based on my analysis, I have enough to document the trigger path. Let me summarize the key findings:

## Root Cause

The vulnerability is a Use-After-Free in `xmt_resolve_od_links()`. When an object descriptor (OD) is freed via `gf_odf_desc_del()` during XMT parsing (in `xmt_node_end` when a descriptor is outside the scene scope at line 727), the `XMT_ODLink` structure's `od` pointer is not set to NULL. The vulnerable `sink` at line 1644 checks `if (l->od && !l->od->objectDescriptorID)` - if the `od` pointer still references freed memory, this becomes a use-after-free. In the good version, there's a nullification at line 700-701: `if (l->od && l->od == (GF_ObjectDescriptor*)desc) { l->od = NULL; }` that the bad version is missing.

## Trigger Path

1. `bad.c:695` `xmt_node_end` - XML parser callback that processes descriptors; calls `gf_odf_desc_del(desc)` at line 727 to free descriptors outside scene scope, including IOD objects
2. `bad.c:1731` `load_xmt_run` - Calls `xmt_resolve_od_links(parser)` after XML parsing completes (line 1731)
3. `bad.c:1644` `xmt_resolve_od_links` - The sink: checks `if (l->od && !l->od->objectDescriptorID)` where `l->od` references freed memory due to missing null assignment after free
