

## Root Cause

The vulnerability is a **Use-After-Free** (CWE-416) in the SANM video decoder. The `stored_frame` buffer (`uint8_t*`) in `SANMVideoContext` stores frame data for STOR/FTCH operations. The vulnerability occurs when:

1. A prior frame processing has stored data to `ctx->stored_frame` (via STOR chunk at lines 568-570 or 576)
2. A subsequent FOBJ chunk triggers `process_frame_obj()` which may call `init_sizes()` → `init_buffers()` (lines 730-731) to resize buffers
3. If `init_buffers()` fails (returns non-zero at line 731), it calls `destroy_buffers()` (line 297) which frees `ctx->stored_frame` via `av_freep()` at line 278 and sets the pointer to NULL
4. Then when processing a later FTCH chunk in the same decode cycle, `process_ftch()` is called (line 589) which accesses `ctx->stored_frame` at line 461 - **this is the use-after-free**, accessing memory that was freed in step 3

The root cause is that `stored_frame` is freed during buffer reinitialization but can still be accessed later in the same frame decode cycle if FTCH processing occurs after a failed reallocation attempt.

## Trigger Path

From analyzing the code flow in `bad.c`:

1. `bad.c:512` `decode_frame(AVCodecContext *avctx, AVFrame *frame, int *got_frame_ptr, AVPacket *pkt)`
   - Entry point for decoding SANM video packets
   - Processes chunks in a loop, handles STOR/FOBJ/FTCH chunks

2. `bad.c:553` `process_frame_obj(SANMVideoContext *ctx, GetByteContext *gb)` (called at line 553)
   - Called when processing FOBJ chunk
   - May trigger buffer resizing if dimensions change

3. `bad.c:730` `init_sizes(SANMVideoContext *ctx, int width, int height)` (called via process_frame_obj)
   - Updates width/height/dimensions in context
   - Called when incoming frame dimensions exceed current buffer dimensions (line 726)

4. `bad.c:731` `init_buffers(SANMVideoContext *ctx)` (called via process_frame_obj)
   - Attempts to allocate/reallocate buffers including stored_frame
   - If allocation fails, calls `destroy_buffers(ctx)` at line 297, **freeing stored_frame at line 278**

5. `bad.c:589` `process_ftch(SANMVideoContext *ctx, int size)` (called at line 589 from decode_frame)
   - **THE SINK**: Accesses freed pointer at line 461 (`uint8_t *sf = ctx->stored_frame`)
   - This is the Use-After-Free vulnerability trigger

The trigger path requires: STOR stores data → later FOBJ triggers reinit that fails → destroy_buffers frees stored_frame → subsequent FTCH in same decode loop accesses the now-freed pointer.
