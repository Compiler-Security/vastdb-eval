## Root Cause

The vulnerability is a **Use After Free (CWE-416)**. The `AVStream *st` object is freed inside `ff_rm_read_mdpr_codecdata` at line 640 when `mime` equals "logical-fileinfo" via `ff_free_stream(s, st)`. However, the caller at line 843 in `rm_read_header` still uses this freed stream object after `rm_read_multi` returns. The `mime` string pointer flows from the caller (line 843) through `rm_read_multi` (line 748) into `ff_rm_read_mdpr_codecdata`, where the condition at line 638 triggers the free. After the function returns, the caller attempts to use the now-freed `st` object, leading to the use-after-free.

## Trigger Path

1. `bad.c:757` `rm_read_header`
   - Entry point that parses RM file header. Declares local `char mime[128]` and passes it to `rm_read_multi` at line 843.

2. `bad.c:843` `rm_read_header` (inline call to `rm_read_multi`)
   - Calls `rm_read_multi(s, s->pb, st, mime)` where `st` is the AVStream created earlier. The `mime` string is passed as the last argument.

3. `bad.c:715` `rm_read_multi`
   - Receives `mime` parameter and passes it to `ff_rm_read_mdpr_codecdata` at line 748 along with `st2` (which equals `st` for i==0).

4. `bad.c:748` `rm_read_multi` (inline call to `ff_rm_read_mdpr_codecdata`)
   - Calls `ff_rm_read_mdpr_codecdata(s, s->pb, st2, st2->priv_data, size2, mime)` with the stream and mime.

5. `bad.c:638` `ff_rm_read_mdpr_codecdata`
   - The sink checks: `if(mime && !strcmp(mime, "logical-fileinfo"))`. When this condition is true, it proceeds to free the stream at line 640.

6. `bad.c:640` `ff_rm_read_mdpr_codecdata`
   - Calls `ff_free_stream(s, st)` which **frees the stream object**. This is where the stream becomes freed.

7. `bad.c:843` (after return) `rm_read_header`
   - Uses the `st` object after `rm_read_multi` returns, but `st` has already been freed at step 6 - **Use After Free triggered**.
