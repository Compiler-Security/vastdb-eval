## Root Cause

The vulnerability is a **Use After Free** in `gf_filter_pid_set_property_full` (bad.c:1161). The issue arises when `PROP_STRING(segext)` is used to set the `GF_PROP_PID_FILE_EXT` property. The `PROP_STRING` macro uses a static global `_pv_str` (bad.c:149) whose address is stored in the property. When the property is set again with the same value, `gf_props_equal_strict` returns true at line 1159, triggering `gf_free(value->value.string)` at line 1161 to free the string. However, the static `_pv_str` is reused for subsequent operations, leading to a use-after-free when the now-freed pointer is accessed. The sink at line 399 (`!strcmp(segext, "m3u8")`) is not the direct trigger but rather the condition that ensures the "m3u8" string is passed to the property setter, leading to the vulnerable code path in `gf_filter_pid_set_property_full`.

## Trigger Path

1. `bad.c:250` `dasher_configure_pid`
   - Entry point for filter PID configuration. At lines 377-378, sets `segext = "m3u8"` based on conditions.

2. `bad.c:384` `gf_filter_pid_set_property(opid, GF_PROP_PID_FILE_EXT, PROP_STRING(segext))`
   - Sets the FILE_EXT property using PROP_STRING macro which uses static global `_pv_str`.

3. `bad.c:1139` `gf_filter_pid_set_property`
   - Wrapper that calls `gf_filter_pid_set_property_full`.

4. `bad.c:1161` `gf_filter_pid_set_property_full`
   - When the same property value is set twice with `GF_PROP_STRING_NO_COPY` type and values are equal, it calls `gf_free(value->value.string)`, freeing the string. Subsequent uses of the freed pointer cause the use-after-free.

The **sink** at bad.c:399 (`!strcmp(segext, "m3u8")`) is a condition that ensures the "m3u8" string flows through to trigger the vulnerable code path, but it's not the direct cause of the use-after-free—the vulnerability is in the property handling logic at line 1161.
