

Looking at the code, I can now trace the vulnerability:

## Root Cause

The vulnerability is a **Use-After-Free** in the `dasher_configure_pid` function. When `gf_filter_pid_caps_query` retrieves a property (e.g., `GF_PROP_PID_FILE_EXT`) that has type `GF_PROP_STRING_NO_COPY`, the string pointer is owned by the filter. Later, when `gf_filter_pid_set_property_full` is called with this property value, it frees the string at line 1161 (`gf_free(value->value.string)`) because the type is `NO_COPY`. However, `segext` still holds the freed pointer from line 368 (`segext = p->value.string`). When `strcmp(segext, "m3u8")` is called at line 399, it accesses the dangling pointer, triggering the use-after-free.

## Trigger Path

1. `bad.c:250` `dasher_configure_pid`
   - Entry point: Called to configure a filter PID. Queries the FILE_EXT property at line 365.

2. `bad.c:365` `gf_filter_pid_caps_query`
   - Queries `GF_PROP_PID_FILE_EXT` from the input PID, returns property `p` which may have type `GF_PROP_STRING_NO_COPY`.

3. `bad.c:367-368` 
   - Calls `gf_filter_pid_set_property(opid, GF_PROP_PID_FILE_EXT, p)` and assigns `segext = p->value.string`.

4. `bad.c:1142` `gf_filter_pid_set_property_full`
   - This function processes the property. At lines 1160-1161, if `value->type==GF_PROP_STRING_NO_COPY`, it calls `gf_free(value->value.string)`, freeing the string that `segext` points to.

5. `bad.c:399` `strcmp(segext, "m3u8")`
   - **Sink**: Uses the freed pointer `segext`, triggering the use-after-free.
