## Root Cause

A `tuple_T` object is allocated in `eval_tuple` at line 1328, and its pointer is stored in the local variable `tuple`. When processing tuple items, the first item from `rettv` is added to the tuple via `tuple_append_tv(tuple, rettv)` at line 1335. If `tuple_append_tv` returns FAIL (line 1336), the function returns FAIL without freeing the allocated `tuple`, causing a memory leak. However, the more critical issue is when the code proceeds to `failret:` label: at line 1412, `tuple_free(tuple)` frees the tuple memory, but the original `rettv` still contains pointers to the data that was stored in the tuple (data from line 1335). After the tuple is freed, any subsequent use of `rettv` accesses freed memory,_triggering the use-after-free.

## Trigger Path

1. `bad.c:1310` `eval_tuple`
   - Receives `rettv` and `evalarg`; allocates a tuple at line 1328; adds first item to tuple via `tuple_append_tv` at line 1335; if processing fails at lines 1353-1367, goto `failret` which calls `tuple_free(tuple)` at line 1412, freeing the tuple while `rettv` may still hold references to its data.

2. `bad.c:283` or `bad.c:292` (from `eval9_nested_expr`)
   - Calls `eval_tuple(arg, rettv, evalarg, TRUE)` with the same `rettv` that was passed into `eval9_nested_expr`; control flows back to caller with potentially invalidated `rettv`.

3. `bad.c:1278` (from `eval9`)
