## Root Cause

In `dwa_uncompress()`, the value `dc_count` is read directly from untrusted compressed input at line 1087 using `AV_RL64(src + 72)` without any bounds validation. This value is then used at line 1135 to compute `dest_len = dc_count * 2LL`, which determines the buffer size allocated at line 1141 via `av_fast_padded_malloc(&td->dc_data, &td->dc_size, FFALIGN(dest_len, 64) * 2)`. While a check exists at line 1138 (`dc_count > (6LL * td->xsize * td->ysize + 63) / 64`), the multiplication can overflow when `td->xsize * td->ysize` is large, causing the check to wrap around and pass when it should fail, leading to an undersized buffer allocation. The CWE-129 array index vulnerability manifests when the buffer is later accessed via the computed index `dc[idx]` at line 1188, where `idx` can exceed the allocated buffer bounds.

## Trigger Path

1. `bad.c:570` `decode_frame` - External entry point; calls `decode_block` via thread pool at line 737
2. `bad.c:756` `decode_block` - Worker for each scanline/tile; calls `dwa_uncompress` at line 918 when compression is DWA
3. `bad.c:1061` `dwa_uncompress` - Reads `dc_count` from untrusted input at line 1087 (source), allocates buffer at line 1141 (sink)
