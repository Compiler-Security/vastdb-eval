## Root Cause

The vulnerability is a null pointer dereference in `PBMAC1_PBKDF2_HMAC` at line 436. The function retrieves the salt from `pbkdf2_param->salt->value.octet_string` at line 421 without checking if `pbkdf2_param->salt` is NULL. In PKCS#12, the salt field in `PBKDF2PARAM` is optional (it's an `ASN1_TYPE *salt`). When the salt is absent, `pbkdf2_param->salt` is NULL, and dereferencing it causes a null pointer dereference when accessing `pbkdf2_salt->data` in the call to `PKCS5_PBKDF2_HMAC`.

## Trigger Path

1. `bad.c:583` `PKCS12_verify_mac`
   - This is the entry point that verifies MAC on a PKCS12 structure. It receives the PKCS12 object and password.

2. `bad.c:611` `pkcs12_gen_mac`
   - Called from `PKCS12_verify_mac` to generate the MAC for verification. Passes the `macalg` and other parameters to `PBMAC1_PBKDF2_HMAC`.

3. `bad.c:517` `PBMAC1_PBKDF2_HMAC`
   - Called when `pbmac1_md_nid != NID_undef && pkcs12_key_gen == NULL`. It extracts the salt from `pbkdf2_param->salt->value.octet_string` at line 421 without validating that `pbkdf2_param->salt` is non-NULL.

4. `bad.c:436` `PKCS5_PBKDF2_HMAC`
   - The vulnerable call uses `pbkdf2_salt->data` and `pbkdf2_salt->length`. When `pbkdf2_param->salt` is NULL (i.e., salt is absent in the PBKDF2PARAM), `pbkdf2_salt` becomes NULL, and dereferencing it triggers the null pointer dereference (CWE-476).
