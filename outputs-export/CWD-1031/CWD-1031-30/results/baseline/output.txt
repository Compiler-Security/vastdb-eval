

## Root Cause

The vulnerability is a NULL pointer dereference (CWE-476) in the variable handling code. The `struct varinfo` pointer `var` is allocated using `bfd_zalloc` (line 744), which zeros all fields including the `file` pointer (initially NULL). When processing a `DW_AT_specification` attribute, the code looks up `spec_var` by offset at line 850. If `spec_var` is found but its `file` field was never set (e.g., the specification variable lacks a `DW_AT_decl_file` attribute), then `spec_var->file` remains NULL. At line 864, `strdup(spec_var->file)` is called with a NULL pointer, causing a null pointer dereference.

## Trigger Path

1. `bad.c:649` `scan_unit_for_symbols`
   - Iterates through DWARF debug information, processing each variable DIE. When encountering a variable with `DW_AT_specification`, it sets up the vulnerability chain.

2. `bad.c:845` `DW_AT_specification` case in `scan_unit_for_symbols`
   - The attribute handler for `DW_AT_specification` is entered. Control flows to the vulnerable statement at line 864.

3. `bad.c:850` `lookup_var_by_offset(attr.u.val, unit->variable_table)`
   - Looks up the specification variable by offset. Returns a `spec_var` pointer that may have uninitialized `file` field.

4. `bad.c:864` `var->file = strdup(spec_var->file)`
   - The vulnerable sink: `strdup(NULL)` is called when `spec_var->file` is NULL (never assigned), triggering the null pointer dereference.
