

## Root Cause

The vulnerability is a NULL pointer dereference (CWE-476). In the `sk_sum_t` struct, fields like `md5`, `sha1`, and `sha256` are character pointers. When `memset(&oldsum, 0, sizeof(sk_sum_t))` is called at line 691, it zeros out the entire struct including the `md5` pointer, making it NULL. The code at line 510 attempts to dereference `newsum->md5` and `oldsum->md5` using `!*newsum->md5` to check if the string is empty. If either pointer is NULL (due to memset zeroing or failed parsing), dereferencing it causes a null pointer dereference.

## Trigger Path

1. `bad.c:673` `fim_db_search`
   - Entry point that receives `oldsum` and `newsum` as stack-allocated structs; calls initialization and decoding functions.

2. `bad.c:691` `memset(&oldsum, ...)` and `bad.c:692` `memset(&newsum, ...)`
   - Zero-initializes both `sk_sum_t` structs, setting all pointer fields including `md5` to NULL.

3. `bad.c:724` `sk_decode_extradata(&oldsum, check_sum)`
   - Decodes old checksum from database into `oldsum`; may partially populate fields or leave some pointers NULL if parsing fails.

4. `bad.c:731` `sk_decode_sum(&newsum, c_sum, w_sum)`
   - Decodes new checksum into `newsum`; may fail to allocate/parse the `md5` field, leaving it NULL.

5. `bad.c:874` `fim_alert(f_name, &oldsum, &newsum, ...)`
   - Passes structs to `fim_alert` for comparison; includes pointers that may still be NULL.

6. `bad.c:510` `fim_alert` - NULL pointer dereference
   - The check `!*newsum->md5 || !*oldsum->md5` dereferences pointers that may be NULL from memset or failed parsing, triggering the vulnerability.
