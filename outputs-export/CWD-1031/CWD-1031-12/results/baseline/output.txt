## Root Cause

The vulnerability is a NULL pointer dereference in `raw_mantissa[c][i]` at line 921. The `raw_mantissa` array is only allocated when `sconf->floating` is true (lines 751-776 in `decode_init`). However, the `read_diff_float_data` function (line 812) that accesses `raw_mantissa[c][i]` at line 921 can be called when `sconf->floating` was true during `decode_init`, but the access at line 921 happens inside a conditional at line 918 that checks `if (!get_bits1(gb))`. The bug is that when the floating config is true but certain conditions in the bitstream lead to access patterns where the per-channel `raw_mantissa[c]` pointer may not be properly initialized or the `nbits[i]` value used as an index could be out of bounds/corrupted, causing access to an invalid address. When `raw_mantissa[c]` is NULL or invalid, dereferencing it at `raw_mantissa[c][i]` triggers NULL pointer dereference.

## Trigger Path

1. `bad.c:992` `decode_frame`
   - Entry point called to decode an audio frame. Sets up bitstream context and calls `read_frame_data`.

2. `bad.c:473` `read_frame_data`
   - Decodes frame data including channel blocks. At line 617, checks `sconf->floating` and calls `read_diff_float_data`.

3. `bad.c:812` `read_diff_float_data`
   - Handles float decoding. At line 918, accesses `raw_mantissa[c][i]` using a 2D array index. **This is the vulnerable sink operation at line 921.**

The trigger path is: `decode_frame` → `read_frame_data` → `read_diff_float_data` → vulnerable access at line 921.
