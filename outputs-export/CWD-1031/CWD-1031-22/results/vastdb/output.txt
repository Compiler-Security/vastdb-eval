## Root Cause

The vulnerability is a **NULL pointer dereference** in `display_relocations()` at bad.c:1318. The function receives a `section` pointer and directly accesses `section->sh_type` without checking if `section` is NULL. When `process_section_headers()` fails to load section headers (e.g., due to memory allocation failure at bad.c:496), `filedata->section_headers` remains NULL or invalid. However, `process_relocs()` is still called (at bad.c:1673) and iterates through the NULL pointer in a for-loop (at bad.c:1574-1578), passing each invalid section pointer to `display_relocations()`, which then dereferences the NULL pointer at line 1318.

## Trigger Path

1. `bad.c:1614` `process_object (Filedata * filedata)`
   - Top-level function that processes ELF files; calls `process_relocs()` at line 1673 regardless of whether section headers loaded successfully.

2. `bad.c:1441` `process_relocs (Filedata * filedata)`
   - Called unconditionally after section header processing. If `do_reloc` is true, iterates through sections at lines 1574-1580 using `filedata->section_headers` which may be NULL.

3. `bad.c:1314` `display_relocations (Elf_Internal_Shdr * section, ...)`
   - Called for each section in the loop. At line 1318, directly accesses `section->sh_type` without NULL check:
   ```c
   relocation_type rel_type = rel_type_from_sh_type (section->sh_type);
   ```
   - When `section` is NULL (due to failed section header allocation), this dereferences a NULL pointer.
