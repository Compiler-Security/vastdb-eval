

## Root Cause

The vulnerability is a NULL pointer dereference (CWE-476). In `ossl_bsearch()` (line 254-255), when the binary search comparison returns non-zero (no match found) and the `OSSL_BSEARCH_VALUE_ON_NOMATCH` flag is NOT set, the pointer `p` is explicitly set to NULL. This NULL value propagates through the call chain from `ssl3_get_cipher_by_id()` → `ssl3_get_cipher_by_char()` → `ossl_quic_get_cipher_by_char()`, and at line 321, `ciph` is dereferenced without a NULL check: `if ((ciph->algorithm2 & SSL_QUIC) == 0)`, causing a NULL pointer dereference.

## Trigger Path

1. `bad.c:318` `ossl_quic_get_cipher_by_char`
   - Receives NULL from `ssl3_get_cipher_by_char()` at line 319

2. `bad.c:312` `ssl3_get_cipher_by_char`
   - Calls `ssl3_get_cipher_by_id()` at line 313, returns whatever that returns

3. `bad.c:297` `ssl3_get_cipher_by_id`
   - Calls `OBJ_bsearch_ssl_cipher_id()` at lines 302, 305, 308 to search cipher arrays

4. `bad.c:232` `ossl_bsearch` (source of NULL)
   - At line 254-255: when `c != 0 && !(flags & OSSL_BSEARCH_VALUE_ON_NOMATCH)`, sets `p = NULL` and returns it
