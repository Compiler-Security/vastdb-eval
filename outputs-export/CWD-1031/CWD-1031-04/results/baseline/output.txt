

## Root Cause

The vulnerability is a NULL pointer dereference in the SSL/TLS cipher lookup chain. In `ossl_bsearch` (crypto/bsearch.c:254-255), when a key is not found in the binary search (`c != 0`) and the `OSSL_BSEARCH_VALUE_ON_NOMATCH` flag is not set, the pointer `p` is explicitly set to NULL and returned. When external input (a cipher character string) flows through the call chain to `ossl_quic_get_cipher_by_char` (ssl/quic/quic_impl.c:318-321), it receives a potentially NULL cipher pointer from `ssl3_get_cipher_by_char` without checking for NULL, then immediately dereferences it at line 321 to access `ciph->algorithm2`. This is a classic CWE-476 NULL pointer dereference where the sink assumes the value is always valid.

## Trigger Path

1. `ssl/quic/quic_impl.c:318` `ossl_quic_get_cipher_by_char`
   - External entry point that receives an unsigned char pointer from network/invoke; calls `ssl3_get_cipher_by_char` and dereferences result without NULL check.

2. `ssl/s3_lib.c:312` `ssl3_get_cipher_by_char`
   - Constructs a cipher ID from the character bytes and calls `ssl3_get_cipher_by_id`.

3. `ssl/s3_lib.c:297` `ssl3_get_cipher_by_id`
   - Performs binary searches on cipher tables using `OBJ_bsearch_ssl_cipher_id` macro; may return NULL when cipher is not found.

4. `crypto/objects/obj_dat.c:265` `OBJ_bsearch_` (via macro wrapper)
   - Wrapper around `OBJ_bsearch_ex_` with default flags (0), passes through the result.

5. `crypto/objects/obj_dat.c:271` `OBJ_bsearch_ex_`
   - Calls `ossl_bsearch` and returns whatever pointer it gets.

6. `crypto/bsearch.c:233` `ossl_bsearch`
   - Binary search implementation; at lines 254-255, sets `p = NULL` when key not found and flag not set: `if (c != 0 && !(flags & OSSL_BSEARCH_VALUE_ON_NOMATCH)) p = NULL;`
