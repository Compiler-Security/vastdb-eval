

## Root Cause

The vulnerability is a NULL pointer dereference in `mov_read_trak` (line 861). The `MOVStreamContext` structure contains two separate fields: `tts_data`/`tts_count` (lines 268-270) and `stts_data`/`stts_count` (lines 271-273). When `mov_read_stts` parses the STTS atom, it populates only `sc->stts_data` and `sc->stts_count` (the "stts" variant). The "tts" variant (`sc->tts_data` and `sc->tts_count`) remains uninitialized at NULL/0 from `av_mallocz` in line 759. Later in `mov_read_trak` (line 860-861), the code incorrectly uses `sc->tts_count` and `sc->tts_data` (without the "stts" prefix) in a loop that dereferences `sc->tts_data[i]`, causing a NULL pointer dereference when `tts_data` is NULL and `tts_count` is 0.

## Trigger Path

1. `bad.c:750` `mov_read_trak`
   - Creates a new AVStream and allocates `MOVStreamContext` via `av_mallocz`, initializing `tts_data` to NULL and `tts_count` to 0
   - Calls `mov_read_default` to parse child atoms including STTS

2. `bad.c:770` `mov_read_default` (via parse table)
   - Dispatches to `mov_read_stts` for STTS atom parsing

3. `bad.c:630` `mov_read_stts`
   - Populates `sc->stts_data` and `sc->stts_count` with parsed STTS entries
   - Does NOT populate `sc->tts_data` or `sc->tts_count`

4. `bad.c:860` `mov_read_trak`
   - After returning from `mov_read_default`, attempts to iterate using `sc->tts_count` and `sc->tts_data` which are still NULL/0
   - Accesses `sc->tts_data[i].duration` and `sc->tts_data[0].duration` at line 861, triggering NULL pointer dereference when `tts_data` is NULL
