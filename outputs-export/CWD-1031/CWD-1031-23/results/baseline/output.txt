

## Root Cause

The vulnerability occurs because `r_io_desc_get` (line 516-518) returns a pointer obtained from `r_id_storage_get`, which returns NULL when the given file descriptor `fd` is not registered in the `io->files` storage (either out of bounds or the slot is empty). At line 569 in the `load` function, the code dereferences this pointer without a NULL check: `bf->rbin->iob.desc_get (bf->rbin->iob.io, bf->fd)->plugin->meta.name`. When `desc_get` returns NULL, accessing `->plugin->meta.name` dereferences NULL, causing CWE-476 (Null Pointer Dereference).

## Trigger Path

1. `bad.c:565` `load(RBinFile *bf, RBuffer *buf, ut64 loadaddr)`
   - The entry function that attempts to load a bin file; calls `desc_get` at line 569 without checking for NULL return.

2. `bad.c:569` (within `load`)
   - Calls `bf->rbin->iob.desc_get(bf->rbin->iob.io, bf->fd)` which is bound to `r_io_desc_get` at line 527; passes the result directly to member access without NULL validation.

3. `bad.c:516` `r_io_desc_get(RIO *io, int fd)`
   - Returns the result of `r_id_storage_get(&io->files, fd)` at line 518; may return NULL if fd is invalid.
