## Root Cause

The vulnerability stems from the `text_value` field of the `Dwg_Entity_TEXT` structure potentially being NULL when `htmlescape()` is called. In `dwg_decode_TEXT_private()` (line 2234), `text_value` is assigned using `bit_read_TV()`, which returns NULL on buffer overflow conditions (line 2202). The return value from `bit_read_TV()` is not null-checked after assignment, leaving `text->text_value` potentially NULL. Later, in `output_TEXT()` (line 1972), `htmlescape(text->text_value, ...)` is called without verifying that `text->text_value` is non-NULL, leading to a NULL pointer dereference when the function attempts to read from the NULL pointer.

## Trigger Path

1. `bad.c:3676` `main` - Entry point that calls `dwg_read_file()` to load DWG data, then calls `output_SVG()`.
2. `bad.c:3506` `dwg_read_file` - Parses the DWG file and populates `Dwg_Data` structure with objects including TEXT entities.
3. `bad.c:2145` `output_SVG` - Iterates over `block_control.entries` and calls `output_BLOCK_HEADER()` for each block.
4. `bad.c:2085` `output_BLOCK_HEADER` - Iterates over block entities via `get_first_owned_entity()` / `get_next_owned_entity()` loop, calling `output_object()` for each.
5. `bad.c:2006` `output_object` - Dispatches to entity-specific output functions based on `obj->type`; for `DWG_TYPE_TEXT` calls `output_TEXT()`.
6. `bad.c:1958` `output_text` - Calls `htmlescape(text->text_value, ...)` at line 1972 without NULL-checking `text->text_value`, causing the NULL pointer dereference.
