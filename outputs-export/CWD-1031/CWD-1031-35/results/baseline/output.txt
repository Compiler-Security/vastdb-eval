

## Root Cause

The vulnerability is a NULL pointer dereference in the `DetectEntropyDoMatch` function at line 1322. The vulnerable object is `det_ctx->p`, a `Packet *` member in `DetectEngineThreadCtx`. At line 2699 (the source), `det_ctx->p` is explicitly set to NULL after transaction inspection completes. At the sink (line 1322), the code dereferences `det_ctx->p->flow` without any NULL check: `FlowVarAddFloat(det_ctx->p->flow, ded->fv_idx, entropy);`. The vulnerability is triggered when `DetectEntropyDoMatch` executes after `det_ctx->p` has been set to NULL during the transaction cleanup phase, causing a dereference of a NULL pointer.

## Trigger Path

1. `bad.c:2893` `DetectFlow`
   - External/top-level entry point for detection, receives Packet `p` and initiates the detection pipeline.

2. `bad.c:2750` `DetectRun`
   - Main detection run function. Sets up detection context and calls `DetectRunTx` at line 2830 to inspect transactions.

3. `bad.c:2351` `DetectRunTx`
   - Transaction inspection loop function. Sets `det_ctx->p = p` at line 2398 during iteration, then sets `det_ctx->p = NULL` at line 2699 after completing all transactions. Calls `DetectRunTxInspectRule` at line 2615 for each rule match.

4. `bad.c:1239` `DetectRunTxInspectRule`
   - Inspects individual signatures against transactions. Calls `DetectEngineContentInspectionInternal` to perform content inspection.

5. `bad.c:1328` `DetectEngineContentInspectionInternal`
   - Performs content inspection on buffer data. When `smd->type == DETECT_ENTROPY`, calls `DetectEntropyDoMatch` at line 1721.

6. `bad.c:1314` `DetectEntropyDoMatch`
   - Computes entropy and writes result using `FlowVarAddFloat(det_ctx->p->flow, ded->fv_idx, entropy)` at line 1322 (sink). This dereferences `det_ctx->p->flow` without checking if `det_ctx->p` is NULL.
