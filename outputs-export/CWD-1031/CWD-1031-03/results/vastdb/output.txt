

## Root Cause

The vulnerability is a NULL Pointer Dereference in the AAC3 audio parser. In `bad.c:383`, `init_get_bits8(&gb, buf, size)` can fail and return an error code when `size` is negative (or too large), which sets the GetBitContext's internal buffer pointer to NULL. However, `avpriv_ac3_parse_header` at line 384 does NOT check this error return value before passing `&gb` to `ff_ac3_parse_header`. When `ff_ac3_parse_header` executes at line 282: `hdr->sync_word = get_bits(gbc, 16)`, it dereferences the NULL buffer, causing a NULL pointer dereference (CWE-476).

## Trigger Path

1. `bad.c:368` `avpriv_ac3_parse_header`
   - Entry point; accepts a `size` parameter from the caller. Does not validate `size` before use.

2. `bad.c:383` `init_get_bits8(&gb, buf, size)`
   - Called with raw `size` value. If `size` is negative or exceeds INT_MAX/8, the function sets `buffer = NULL` and returns error code -22, but this error is NOT checked by the caller.

3. `bad.c:384` `ff_ac3_parse_header(&gb, hdr)`
   - Called with the initialized GetBitContext `gb` WITHOUT checking if `init_get_bits8` succeeded. The GetBitContext may have a NULL buffer.

4. `bad.c:282` `get_bits(gbc, 16)`
   - The sink operation: dereferences `gbc->buffer` to read 16 bits. When `gbc->buffer` is NULL (due to earlier failure), this triggers the NULL pointer dereference.
