## Root Cause

The vulnerability is in `mk_http_range_parse` at lines 339-340. The `mk_string_copy_substr` function can return NULL when:
1. The input string is empty or invalid (causing `pos_init > pos_end` at line 281-283)
2. Memory allocation fails at lines 290-294

In the "=yyy-xxx" case (line 338), when `mk_string_copy_substr(sr->range.data, eq_pos + 1, sep_pos)` returns NULL due to allocation failure or other edge cases, the code directly passes this NULL pointer to `atol(buffer)` at line 340 without any NULL check. This causes a null pointer dereference when `atol` attempts to read from the NULL address.

## Trigger Path

The minimal trigger path to the vulnerability:

1. `bad.c:303` `mk_http_range_parse(struct mk_http_request *sr)`
   - Entry point that parses HTTP Range header. The `sr->range.data` field contains the raw range string. Calls `mk_string_char_search` to find '=' and '-' positions, then calls `mk_string_copy_substr` at line 339.

The test case appears to contain only this single file focused on the vulnerable function. There's no external/main entry point file visible in the glob results - the vulnerability can be triggered by directly calling `mk_http_range_parse` with a crafted `mk_http_request` struct where `range.data` causes `mk_string_copy_substr` to return NULL (e.g., when memory allocation fails or edge case inputs occur).
