

## Root Cause

The vulnerability is a **NULL pointer dereference** in `mk_http_range_parse`. When `mk_string_copy_substr` is called at line 339 with `pos_init > pos_end` (e.g., `eq_pos + 1 > sep_pos`), it returns `NULL` per lines 281-282 of `mk_string_copy_substr`. At line 340, this NULL is passed directly to `atol(buffer)` without a NULL check, causing undefined behavior when attempting to dereference the NULL pointer.

## Trigger Path

1. **bad.c:276** `mk_string_copy_substr`
   - Returns NULL when `pos_init > pos_end` (line 281-282) or on allocation failure (line 292-294)

2. **bad.c:303** `mk_http_range_parse`
   - Entry point. Parses HTTP Range header from `sr->range.data`
   - At line 338-339, calls `mk_string_copy_substr(sr->range.data, eq_pos + 1, sep_pos)` without checking if result is NULL
   - At line 340, passes the result directly to `atol(buffer)` which dereferences it

The function would be triggered by external HTTP request handling code passing a Range header where certain positions create an invalid substring range, causing `mk_string_copy_substr` to return NULL which is then passed to `atol`.
