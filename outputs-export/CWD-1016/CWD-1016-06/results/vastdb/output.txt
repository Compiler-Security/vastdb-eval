## Root Cause

The vulnerability stems from incorrect initialization of the `planeheight` array in `config_input`. At line 555, the code unconditionally sets `planeheight[0] = planeheight[3] = inlink->h` (the full frame height), but for image formats with fewer than 4 planes (e.g., 1 plane for grayscale, 3 for YUV420), plane index 3 does not exist. This incorrectly assigns the full frame height to a non-existent plane, leading to incorrect height values being used when computing iteration bounds in `deinterlace_slice`. When `height` (from `s->planeheight[plane]`) is used to calculate loop bounds at lines 611-612 for the thread job division, an incorrect height value can cause the filter functions to iterate beyond allocated buffer boundaries, resulting in out-of-bounds writes to `work_line` at the sink location (line 537).

## Trigger Path

1. `bad.c:544` `config_input`
   - Initializes `s->planeheight[3] = inlink->h` unconditionally at line 555, even for planes that don't exist in the current pixel format.

2. `bad.c:591` `deinterlace_slice`
   - Reads `height = s->planeheight[plane]` at line 607 and computes job bounds using this potentially incorrect height at lines 611-612.
   - Passes calculated bounds to filter functions.

3. `bad.c:525` `filter16_complex_low` (or similar filter function)
   - The sink location where out-of-bounds write occurs at line 537: `*work_line = *in_lines_cur[0]++ * coef[0]`
   - Uses `work_line` buffer allocated based on `linesize[0]` but iteration count controlled by the potentially incorrect `height` value.
