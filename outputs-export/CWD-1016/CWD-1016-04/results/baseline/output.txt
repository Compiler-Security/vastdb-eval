

## Root Cause

The vulnerability is caused by a mismatch between the width stored in `s->width` and the actual width of the input frame `ref`. At line 403, `s->width` is set to the value `w` passed from `config_input_ref`, which comes from `ctx->inputs[0]->w`. However, at line 497, `convolution_y` uses `s->width` as the iteration bound while reading from `ref->data[0]` and writing to `s->temp_data`. If the caller passes a frame whose actual dimensions (`ref->width`, `ref->linesize[0]`) differ from the initialized `s->width`, the convolution operations can write beyond the bounds of `s->temp_data` (which was allocated based on `s->stride` and `s->height`). This constitutes an out-of-bounds write (CWE-787).

## Trigger Path

1. `bad.c:423` `config_input_ref`
   - Called during filter graph initialization to configure the reference input; extracts dimensions from `inlink` and calls `ff_vmafmotion_init`, setting up `s->width`, `s->height`, and allocating buffers.

2. `bad.c:428` `ff_vmafmotion_init`
   - Stores the input link's dimensions into `s->width` and `s->height` at line 403-404; allocates internal buffers (`blur_data`, `temp_data`) based on these dimensions.

3. `bad.c:531` `filter_frame`
   - Entry point when a new frame arrives from the filter graph; receives the actual input frame `ref` which may have dimensions differing from those used during initialization.

4. `bad.c:518` `do_vmafmotion`
   - Extracts `VMAFMotionContext *s` from filter context and calls `ff_vmafmotion_process` passing `ref`.

5. `bad.c:493` `ff_vmafmotion_process`
   - The vulnerable function: calls `s->vmafdsp.convolution_y` using `s->width` (initialized value) while operating on `ref->data[0]` (actual frame data with its own dimensions), causing out-of-bounds write when they differ.
