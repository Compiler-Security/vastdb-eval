

Based on my analysis:

## Root Cause

The vulnerability is an out-of-bounds write in the `convolution_y` function. The root cause is a **stride mismatch**: In `ff_vmafmotion_init` (line 405), `s->stride` is computed as `FFALIGN(w * sizeof(uint16_t), 32)` = `FFALIGN(w * 2, 32)` (32-byte aligned). But the convolution_y call at line 497-498 passes `ref->linesize[0]` as the source stride instead of `s->stride`. When `ref->linesize[0]` has different alignment or value than the expected 32-byte aligned stride, the memory access `dst[i * dst_stride + j]` at line 457, 466, 479 writes beyond the allocated `s->temp_data` buffer (allocated at line 410 based on `s->stride * h`).

## Trigger Path

1. `bad.c:531` `filter_frame` - entry point, receives AVFrame
2. `bad.c:534` calls `do_vmafmotion(ctx, ref)` - passes frame to process
3. `bad.c:523` `do_vmafmotion` - calls `ff_vmafmotion_process(&s->data, ref)`
4. `bad.c:428` `config_input_ref` - before any frame arrives, initializes `s->data` by calling `ff_vmafmotion_init(&s->data, w, h, fmt)` - sets `s->width = w` and allocates buffers based on `s->stride`
5. `bad.c:403` `ff_vmafmotion_init` - stores width and computes stride (32-byte aligned), allocates buffers
6. `bad.c:497` `ff_vmafmotion_process` - calls `convolution_y` with **stride mismatch**: uses `ref->linesize[0]` instead of `s->stride` for source stride, causing out-of-bounds writes to `s->temp_data` buffer
