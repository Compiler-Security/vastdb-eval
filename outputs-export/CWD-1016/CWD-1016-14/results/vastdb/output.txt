## Root Cause

The vulnerability is an **out-of-bounds write** in the `asn1_type_get_int_oct` function at line 218 in `bad.c`. The function copies data from ASN1-encoded octet strings into fixed-size buffers (`iv` array of size `EVP_MAX_IV_LENGTH` = 16). At line 241, the `max_len` parameter is set to `i`, which is the **actual decoded length from the ASN1 data**, not bounded by the buffer size. When the ASN1-encoded data length exceeds 16 bytes, the `memcpy` at line 199 writes beyond the 16-byte `iv` buffer, causing a heap out-of-bounds write.

## Trigger Path

1. `bad.c:249` `evp_cipher_asn1_to_param_ex`
   - Entry point function handling cipher parameter extraction. At line 284, dispatches to `evp_cipher_get_asn1_aead_params` for GCM mode.

2. `bad.c:228` `evp_cipher_get_asn1_aead_params`
   - Extracts AEAD parameters from ASN1_TYPE. At line 238, first calls `ossl_asn1_type_get_octetstring_int` with `EVP_MAX_IV_LENGTH` (16) to get the encoded length. At line 241, calls the same function again using the **unescaped length** (`i`) as max_len.

3. `bad.c:204` `ossl_asn1_type_get_octetstring_int`
   - Wrapper function that unpacks ASN1 sequence and calls `asn1_type_get_int_oct`. At line 218, passes untrusted length value directly as `max_len`.

4. `bad.c:185` `asn1_type_get_int_oct` (static helper)
   - Performs the vulnerable copy. At lines 193-196, computes `n = (max_len > ret) ? ret : max_len` where `ret` is the actual ASN1 string length and `max_len` is the unescaped value from caller. At line 199, `memcpy(data, ..., n)` writes `n` bytes into a buffer sized only for `EVP_MAX_IV_LENGTH` (16). When `n` > 16, out-of-bounds write occurs.
