## Root Cause

In `kek_unwrap_key`, the vulnerability is a buffer overflow. At line 352 in the caller, `key = OPENSSL_malloc(pwri->encryptedKey->length)` allocates an output buffer sized by the **encrypted** input length. Then inside `kek_unwrap_key`, at line 254, the output length `*outlen = tmp[0]` is set from the **decrypted** data's first byte - an attacker-controlled value. At line 255, `memcpy(out, tmp + 4, *outlen)` copies `*outlen` bytes to the output buffer without validating that `*outlen` fits within the allocated buffer size. If `tmp[0]` exceeds the original encrypted key length, an out-of-bounds write occurs.

## Trigger Path

1. `bad.c:355` `ossl_cms_RecipientInfo_pwri_crypt`
   - Calls `kek_unwrap_key(key, &keylen, ...)` where `key` is allocated with size `pwri->encryptedKey->length` (line 352)

2. `bad.c:206` `kek_unwrap_key`
   - Receives `out` buffer allocated based on encrypted input size
   - At line 254: `*outlen = (size_t)tmp[0]` sets output length from attacker-controlled decrypted byte
   - At line 255: `memcpy(out, tmp + 4, *outlen)` writes out-of-bounds when `tmp[0] > allocated_buffer_size`
