

## Root Cause

The vulnerability stems from a missing bounds check when computing `block_pos_bottom` and `block_pos_right` in the `filter_slice` function. In the BAD version (line 792-793), these values are computed as `height - s->block_size` and `width - s->block_size` without the protective `FFMAX(0, ...)` wrapper that exists in the GOOD version (line 793-794). When video frame dimensions are smaller than the block size, these computed values become negative. Later in the loop (line 802-817), the indices `j` and `i` can propagate these negative values to `block_matching` (line 812) and subsequently to the block filtering functions that call `get_block_row`, causing out-of-bounds array access and write operations (`dst[j] = src[j]` pattern in filtering).

## Trigger Path

1. **bad.c:822** `filter_frame`
   - Top-level entry that sets up frame processing and calls thread executor

2. **bad.c:779** `filter_slice`
   - Computes `block_pos_bottom` and `block_pos_right` without `FFMAX(0, ...)` protection; iterates with j,i indices. Call: `block_matching(s, ref, ref_linesize, j, i, plane, jobnr)` at line 812

3. **bad.c:505** `block_matching` (forward declaration - VAST DB uid 1063)
   - Called with potentially negative j,i values; performs block matching operations

4. **bad.c:814** `s->block_filtering` (function pointer to `basic_block_filtering` at line 528)
   - Calls `s->get_block_row(src, src_linesize, y + i, x, ...)` which lacks boundary validation, leading to out-of-bounds write operations on stack/heap buffers
