

## Root Cause

The vulnerability lies in a dimension mismatch between computed plane width and actual buffer stride. In `config_props`, for chroma planes (p>0), `plane->width` is computed as `AV_CEIL_RSHIFT(inlink->w, hsub)` which accounts for chroma subsampling (typically half the luma width for YUV420P). However, in `filter_frame`, when `gaussian_blur` is called with `in->data[p]` and `in->linesize[p]`, the function uses `width` for bounds checking but accesses memory using `src_linesize` stride. When `src_linesize != width` (common due to video buffer alignment), the access `dst[1] = src[1]` can write/read outside the allocated plane buffer, causing an out-of-bounds write (CWE-787).

## Trigger Path

1. `bad.c:250` `filter_frame`
   - Receives AVFrame with video data; iterates over plane index `p`, retrieves `plane->width` from previously configured EdgeDetectContext, passes to gaussian_blur

2. `bad.c:287` `gaussian_blur` (inline call)
   - Called with `width` (plane->width derived from AV_CEIL_RSHIFT subsampling), uses it as loop bounds but accesses arrays using `src_linesize` stride from AVFrame linesize

3. `bad.c:216` `dst[1] = src[1]`
   - The out-of-bounds write occurs here—accessing `dst` and `src` using implicit stride 1 instead of `dst_linesize`/`src_linesize`, combined with `width` value that doesn't match actual buffer stride
