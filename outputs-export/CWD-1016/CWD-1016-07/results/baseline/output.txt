

## Root Cause

The vulnerability is an out-of-bounds write in the `crossfade_samples` function call at line 448. The AudioFadeContext field `s->nb_samples` (set via filter configuration, e.g., via an option) is used directly as the sample count for iterating and writing output samples without validating that the consumed input frames (`cf[0]` and `cf[1]`) actually contain at least `s->nb_samples`. When fewer samples are available in the queued input frames than `s->nb_samples`, the crossfade loop writes beyond the bounds of the input frame buffers into the output buffer.

## Trigger Path

1. `bad.c:389` `activate`
   - This is the filter's activation function called during filter graph processing. It evaluates conditions on the queued samples at inputs[0] and inputs[1].

2. `bad.c:430` condition check
   - The code checks `ff_inlink_queued_samples(ctx->inputs[1]) >= s->nb_samples` - if true, proceeds to crossfade path. This condition alone doesn't guarantee the frames have exactly `s->nb_samples`, only that at least `s->nb_samples` are queued cumulatively.

3. `bad.c:436` `ff_inlink_consume_samples(ctx->inputs[0], s->nb_samples, s->nb_samples, &cf[0])`
   - Consumes up to `s->nb_samples` from input[0], stores result in `cf[0]`. The returned frame may have fewer samples than requested.

4. `bad.c:442` `ff_inlink_consume_samples(ctx->inputs[1], s->nb_samples, s->nb_samples, &cf[1])`
   - Consumes up to `s->nb_samples` from input[1], stores result in `cf[1]`. The returned frame may have fewer samples than requested.

5. `bad.c:448` `s->crossfade_samples(out->extended_data, cf[0]->extended_data, cf[1]->extended_data, s->nb_samples, ...)`
   - **Sink location**: Calls crossfade_samples using `s->nb_samples` as the loop bound without checking that `cf[0]->nb_samples` and `cf[1]->nb_samples` are >= `s->nb_samples`. When the consumed frames have fewer samples than `s->nb_samples`, the loop writes/reads out of bounds.
