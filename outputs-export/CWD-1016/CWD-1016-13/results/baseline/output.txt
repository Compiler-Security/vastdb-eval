

## Root Cause

The vulnerability is an out-of-bounds write in the `OPENSSL_uni2utf8` function. The variable `asclen` is used as both a length accumulator and as a buffer offset without bounds checking. In the second loop (lines 168-175), the code writes UTF-8 output to `asctmp + asclen` by calling `bmp_to_utf8(asctmp + asclen, uni + i, unilen - i)`, but the length parameter passed to `bmp_to_utf8` is `unilen - i` (the remaining input length), which only limits the input read from the UTF-16 buffer, not the output written to the UTF-8 buffer. Since `bmp_to_utf8` can write up to 4 bytes per call (for 4-byte UTF-8 sequences) and `asclen` is incremented by `j` after each write, there is no check to ensure `asclen + j` stays within the allocated buffer size. This allows writing beyond the allocated `asctmp` buffer, causing an out-of-bounds write.

## Trigger Path

1. `bad.c:135` `OPENSSL_uni2utf8`
   - Entry point that converts UTF-16 to UTF-8; calls `bmp_to_utf8` which internally calls `UTF8_putc` to write UTF-8 bytes

2. `bad.c:104` `bmp_to_utf8`
   - Processes UTF-16 characters and calls `UTF8_putc` to write UTF-8 output; receives `asctmp + asclen` as the output buffer pointer and uses `unilen - i` as length, which does not bound the output writes

3. `bad.c:58` `UTF8_putc`
   - Writes the UTF-8 encoded character to the buffer; based on the Unicode code point value, writes 1-4 bytes sequentially starting at the provided pointer without any boundary validation against the buffer allocation size

4. `bad.c:174` `asclen += j`
   - The vulnerable operation where `asclen` is incremented by `j` (the number of UTF-8 bytes written, up to 4), serving as the offset into the output buffer; without verifying that `asclen + j` remains within the allocated buffer size, this allows the offset to exceed the buffer boundaries, triggering the out-of-bounds write (CWE-787)
