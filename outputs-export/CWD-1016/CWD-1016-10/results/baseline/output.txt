## Root Cause

The vulnerability is in the `gaussian_blur` function at lines 460-461 and 491-492 where `memcpy` uses `w` (the `width` parameter from `plane->width`) as the copy size. However, the actual source data comes from an `AVFrame` with `in->linesize[p]` as the source stride. When `in->linesize[p]` is larger than `width` (which can happen due to memory alignment), using `w` as the copy size copies only `width` bytes, but the pointer arithmetic `src += src_linesize` advances by the full `in->linesize[p]`. After several rows, `src` points beyond the valid image data region, yet subsequent iterations (especially the inner loop processing at lines 467-483) read from arbitrary memory locations outside the image buffer, causing an out-of-bounds read that can corrupt memory or leak data. The root cause is a mismatch between the copy size (`w` = `width`) and the actual source stride (`src_linesize` = `in->linesize[p]`).

## Trigger Path

1. `bad.c:495` `filter_frame(AVFilterLink *inlink, AVFrame *in)`
   - Receives the input video frame `in` from the filter pipeline, extracts plane-specific data like `in->data[p]` and `in->linesize[p]` which may have linesize larger than width.

2. `bad.c:532` `gaussian_blur(ctx, width, height, tmpbuf, width, in->data[p], in->linesize[p])`
   - Passes `in->data[p]` as source buffer and `in->linesize[p]` as source stride. The function uses `width` internally as the copy size, not knowing that source stride differs from width.

3. `bad.c:460` `gaussian_blur` memcpy operation
   - Copies `w` bytes using `memcpy(dst, src, w)` but advances pointers by `src_linesize` (`in->linesize[p]`), causing the source pointer to walk beyond the actual image data when linesize > width.
