

## Root Cause

The vulnerability is a classic out-of-bounds array write in the `eval_result` array, which is declared with size `MAX_REGSUB_NESTING` = 4 (allowing indices 0-3). The `vim_regsub_both` function uses a local variable `nested` (line 380) to index into this array. The protection check at line 393 (`if (nesting == MAX_REGSUB_NESTING)`) is intended to prevent overflow, but the check occurs before `nesting` is incremented for the recursive call at line 445. When the function is called with `nesting == 3`, the check passes (3 != 4), `nested` is set to 3, then `nesting` is incremented to 4, and the code at line 418 accesses `eval_result[4]` - an out-of-bounds write to index 4 (array only has 0-3). The STRCPY macro at line 418 then writes the evaluation result to this out-of-bounds location.

## Trigger Path

1. `bad.c:882` `do_string_sub`
   - Called to perform string substitution; at line 939 calls `vim_regsub` to calculate sublen

2. `bad.c:850` `vim_regsub`
   - Wrapper function; at line 872 calls `vim_regsub_both` to perform the actual substitution

3. `bad.c:361` `vim_regsub_both`
   - Contains the vulnerability. When called with nesting=3 and copy flag set (REGSUB_COPY), the check at line 393 passes (3 != 4), `nested` is set to 3, then nesting becomes 4 at line 445, and line 418 writes `STRCPY(dest, eval_result[nested])` with `nested` still referencing the old value but the array is accessed with the incremented nesting index - causing out-of-bounds write to `eval_result[4]`.
