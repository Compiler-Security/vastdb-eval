## Root Cause

The vulnerability is an improper input validation issue in COFF (Common Object File Format) symbol table handling. The vulnerable value is `x_tagndx.l` in an auxiliary entry (`auxent`). At the known vulnerable location (line 2691), the code sets `auxent->u.auxent.x_sym.x_tagndx.p = table_base + auxent->u.auxent.x_tagndx.l` without validating that `x_tagndx.l` is within bounds of the symbol table. At the sink (line 1850), the code dereferences `x_tagndx.p->offset` without checking if `x_tagndx.p` points to valid memory within the table bounds. If `x_tagndx.l` is crafted to be larger than the table size, `x_tagndx.p` will point outside the allocated symbol table, and dereferencing it causes out-of-bounds memory access.

## Trigger Path

Based on the code analysis:

1. `bad.c:2691` `coff_pointerize_aux` (known vulnerable location - source)
   - Sets `x_tagndx.p = table_base + x_tagndx.l` without bounds checking when `x_tagndx.l > 0`

2. `bad.c:1850` `coff_mangle_symbols` (sink)
   - Dereferences `x_tagndx.p->offset` without validating the pointer is within table bounds

The trigger path involves:
- Loading/slurping a COFF binary symbol table via `coff_get_normalized_symtab` (lines 2697-2750)
- Which calls `coff_pointerize_aux` for each auxiliary entry
- Later, when writing object contents via `coff_write_object_contents` calling `coff_mangle_symbols`
- The vulnerable code accesses the previously set `x_tagndx.p` pointer without validation
