## Root Cause

The vulnerability is in COFF auxiliary symbol handling. In `coff_pointerize_aux()` (line 2691-2692), the code computes a pointer `x_tagndx.p` by adding `table_base` to an unvalidated index value `x_tagndx.l` from the auxiliary entry. Later, in `coff_mangle_symbols()` (line 1850), this pointer is dereferenced via `p->offset` to write back to `x_tagndx.l`. If `x_tagndx.l` contains an out-of-bounds index (from a malicious COFF file), the pointer `p` will be invalid, and dereferencing it triggers undefined behavior/crash. This is CWE-20 (Improper Input Validation) - the index value is not validated as a proper table offset before being used as a pointer.

## Trigger Path

1. `bad.c:1868` `coff_write_object_contents`
   - Calls `coff_mangle_symbols()` at line 2443 to write symbols

2. `bad.c:1812` `coff_mangle_symbols`
   - Iterates over symbols and auxiliary entries
   - At line 1850 (SINK): Dereferences `a->u.auxent.x_sym.x_tagndx.p->offset` to write to `x_tagndx.l`

3. `bad.c:2697` `coff_get_normalized_symtab`
   - Called during symbol reading to normalize the symbol table
   - Calls `coff_pointerize_aux()` at line 2767 for each auxiliary entry

4. `bad.c:2656` `coff_pointerize_aux`
   - At line 2691-2692 (SOURCE): Sets `auxent->u.auxent.x_sym.x_tagndx.p = table_base + auxent->u.auxent.x_sym.x_tagndx.l`
   - Uses unvalidated `l` value from the input file without bounds checking
