

## Root Cause

The vulnerability is in `AudioCrossoverContext` where `s->xover` is allocated via `av_calloc` at line 418 in `config_input`, but the `uninit` function (lines 513-522) only frees `s->splits` (line 518) and never frees `s->xover`. When the filter is destroyed, the memory allocated for `s->xover` is not released, causing a memory leak.

## Trigger Path

The full trigger path cannot be determined from this source file alone, as it requires FFmpeg's filter graph infrastructure to invoke the filter lifecycle functions. However, the code shows:

1. `bad.c:459` `init` 
   - Initializes filter, allocates `s->splits`

2. `bad.c:411` `config_input`
   - Called during filter configuration, allocates `s->xover` at line 418

3. `bad.c:513` `uninit`
   - Called when filter is destroyed, frees `s->splits` but NOT `s->xover` - this is where the leak occurs
