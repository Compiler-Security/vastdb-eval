## Root Cause

In `bad.c`, the `config_input` function at line 418 allocates memory for `s->xover` using `av_calloc(inlink->channels, sizeof(*s->xover))`. This allocates an array of `CrossoverChannel` structures with `inlink->channels` elements. However, the `uninit` function at lines 513-522 only frees `s->splits` (line 518) but never frees `s->xover`. When the filter is destroyed via `uninit`, the `xover` memory is never released, causing a memory leak (CWE-401).

## Trigger Path

I cannot establish a complete call chain from external code to `config_input` via VAST DB because:
- The code uses FFmpeg's runtime function pointer pattern where `config_props` in the `AVFilter` struct is set to point to `config_input`
- This linkage happens in FFmpeg framework code not present in this file
- VAST DB shows MLIR operations only for struct/enum definitions (lines 100-153), not the runtime-resolved function calls

From the source code:

1. `bad.c:459` `init`
   - Allocates `s->splits` at line 465

2. `bad.c:411` `config_input`
   - Called when filter link is configured (via `config_props` function pointer)
   - Allocates `s->xover` at line 418
   - **This is the vulnerable allocation**

3. `bad.c:513` `uninit`
   - Called when filter is destroyed
   - Frees `s->splits` at line 518
   - **Missing: Does NOT free `s->xover`** - this is the bug
