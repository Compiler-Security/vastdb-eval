

## Root Cause

In `ff_frame_thread_encoder_init` (bad.c:1426), multiple `AVCodecContext` objects are allocated via `avcodec_alloc_context3()` at line 1506 for thread workers. These allocations occur inside a loop (lines 1503-1531) where each thread gets its own `thread_avctx`. When any operation in the loop fails (lines 1507, 1512, 1518, 1524, or 1528), the code jumps to the `fail:` label (line 1536-1540). The cleanup function `ff_frame_thread_encoder_free(avctx)` at line 1539 only frees the parent `ThreadContext` structure, but does NOT free the individual `thread_avctx` objects allocated at line 1506. The `thread_avctx` pointers are stored in local variables that go out of scope when the function returns -1, causing a memory leak.

## Trigger Path

1. `bad.c:1506` `avcodec_alloc_context3(avctx->codec)` - Allocates a new AVCodecContext for a thread worker
2. `bad.c:1524` `avcodec_open2(thread_avctx, avctx->codec, NULL)` - Opens the codec; may fail
3. `bad.c:1539` `ff_frame_thread_encoder_free(avctx)` - Called on fail path
4. `bad.c:1540` `return -1` - Returns error without freeing thread_avctx memory

The sink is line 1540 where the function returns -1 on the failure path. The vulnerability triggers when an error condition causes early return at line 1540, leaving the `thread_avctx` objects from line 1506 unfreed.
