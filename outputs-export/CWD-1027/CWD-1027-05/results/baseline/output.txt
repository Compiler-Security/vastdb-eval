## Root Cause

A `GF_PropertyMap` is allocated in `check_new_pid_props()` (line 799) via `gf_props_new()`, which calls `GF_SAFEALLOC(map, GF_PropertyMap)` at line 842. This allocates memory for the property map and sets `reference_count = 1` at line 852. The allocated map becomes `dst_props` in `gf_filter_pid_merge_properties_internal()`. At line 907, `gf_props_merge_property(dst_props, src_props, ...)` is called, which returns `GF_OK` at line 952. However, after successful merge, the newly allocated `dst_props` is never freed—the reference count is never decremented, and `gf_props_del()` is never called. The memory leak occurs because the caller returns success without releasing the allocated map.

## Trigger Path

1. `bad.c:856` `gf_filter_pid_merge_properties_internal`
   - Calls `check_new_pid_props()` at line 869 to get/create `dst_props`

2. `bad.c:799` `check_new_pid_props`
   - Calls `gf_props_new()` to allocate a new `GF_PropertyMap` when no old map exists

3. `bad.c:835` `gf_props_new`
   - Uses `GF_SAFEALLOC` at line 842 to allocate `GF_PropertyMap`, sets `reference_count = 1` at line 852, returns the map

4. `bad.c:907` `gf_filter_pid_merge_properties_internal`
   - Calls `gf_props_merge_property(dst_props, src_props, ...)` to merge properties into the newly allocated map

5. `bad.c:952` `gf_props_merge_property`
   - Returns `GF_OK` without freeing `dst_props`, causing the memory leak. The allocated map is added to the properties list at `bad.c:803`, but after merge completes, there is no corresponding release—the `reference_count` remains at 1 and `gf_props_del()` is never invoked.
