## Root Cause

The vulnerability is a **Missing Release of Memory after Effective Lifetime (CWE-401)**. The `streamIterator` structure contains an embedded `raxIterator` which internally uses a `raxStack`. When `streamIteratorStart()` is called at line 423, it calls `raxStart()` (line 394) which calls `raxStackInit()` (line 366). The `raxStack` can dynamically allocate heap memory when it overflows its static buffer (using `realloc`), but there is no corresponding cleanup function to free this allocated memory when the iterator goes out of scope. When `streamGetEdgeID()` returns at line 430, the stack-allocated `streamIterator si` is destroyed without freeing any dynamically allocated memory in its `raxIterator.raxStack.stack` field.

## Trigger Path

1. `bad.c:419` `streamGetEdgeID`
   - Declares local stack variable `streamIterator si` and calls `streamIteratorStart()`

2. `bad.c:423` `streamIteratorStart`
   - Calls `raxStart(&si->ri, s->rax)` to initialize the internal radix tree iterator

3. `bad.c:394` `raxStart` (operation uid 470 → function 226)
   - Calls `raxStackInit(&it->stack)` to initialize the stack structure

4. `bad.c:366` `raxStackInit` (operation uid 470)
   - Initializes `ts->stack = ts->static_items`, but if the stack overflows, heap memory may be allocated via `realloc` without a corresponding cleanup

5. `bad.c:430` `return;`
   - Function returns with `si` going out of scope; any heap-allocated stack memory is never freed

The trigger path shows that calling `streamIteratorStart(&si, s, NULL, NULL, !first)` can trigger dynamic memory allocation in the internal `raxStack`, and the function returns without freeing that memory.
