## Root Cause

The vulnerability is a **Missing Release of Memory after Effective Lifetime (CWE-401)**. In the `filter_frame` function (bad.c:805), a new output frame `out` is allocated via `ff_get_video_buffer(outlink, outlink->w, outlink->h)` at line 813. The `out` frame is passed to `ff_filter_frame(outlink, out)` at line 913, which transfers ownership to the next filter via reference counting. However, the input frame `in` is freed at line 911 with `av_frame_free(&in)`, but `out` is **never explicitly freed** in any code path. If errors occur after line 813 (e.g., in `create_filtergraph` at line 889, or in the processing at lines 903-910), the `out` frame is never released before returning an error, causing a memory leak.

## Trigger Path

1. `bad.c:805` `filter_frame(AVFilterLink *link, AVFrame *in)`
   - Entry point function that processes video frames in the colorspace filter

2. `bad.c:813` `ff_get_video_buffer(outlink, outlink->w, outlink->h)` (via `hl.call` uid 2778)
   - Allocates a new AVFrame buffer `out` with reference count = 1

3. `bad.c:889` `create_filtergraph(ctx, in, out)` or subsequent processing
   - Multiple error paths exist after allocation; on error, function returns without freeing `out`

4. `bad.c:911` `av_frame_free(&in)`
   - Only frees the input frame `in`, not the output frame `out`

5. `bad.c:913` `ff_filter_frame(outlink, out)`
   - Transfers `out` to next filter; if this fails or earlier error occurs, `out` is never freed

The leak occurs because there is no `av_frame_free(&out)` call anywhere in the function - the code incorrectly assumes ownership transfer to the next filter always succeeds, but any error path before or during `ff_filter_frame` leaves `out` unfreed.
