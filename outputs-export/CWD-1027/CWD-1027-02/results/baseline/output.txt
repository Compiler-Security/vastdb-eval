## Root Cause

The vulnerability is in the `tile` filter's `uninit` function at line 275-280. The `TileContext` structure contains two frame pointers: `out_ref` (line 169) and `prev_out_ref` (line 170). When `out_ref` is first NULL at line 229, a new frame is allocated via `ff_get_video_buffer(outlink, outlink->w, outlink->h)` at line 230. However, in the `uninit` function, only `prev_out_ref` is freed (line 279), while `out_ref` is never released. This causes a memory leak where the AVFrame buffer allocated at line 230 is never freed after the filter's effective lifetime ends.

## Trigger Path

I cannot prove a complete function call stack from the external entry point to the vulnerable operation based solely on this single-file test case. The code shows:

1. `bad.c:222` `filter_frame`
   - Called when a frame arrives at the filter; checks if `tile->out_ref` is NULL at line 229, and if so, allocates a new frame buffer at line 230.

The filter appears to be part of an FFmpeg filter graph, but there is no explicit call site in this file that invokes `filter_frame` or triggers the uninit path. The test case does not include the top-level entry point (e.g., `main()`) that would demonstrate how frames are fed into the filter or how cleanup is triggered.
