

## Root Cause

The vulnerability is a memory leak in the tile video filter. The `TileContext` struct contains two `AVFrame*` members: `out_ref` and `prev_out_ref`. In `filter_frame`, the code allocates a new video buffer via `ff_get_video_buffer()` and stores it in `tile->out_ref` (line 230). When the filter isuninitialized, `uninit()` only frees `tile->prev_out_ref` (line 279), leaving `tile->out_ref` unfreed. This causes leaked memory when the filter is unloaded.

## Trigger Path

I was unable to trace a complete call-chain from external entry to `filter_frame` using VAST DB, as this is an internalFFmpeg filter callback set via function pointer (not a direct function call). The trigger requires the FFmpeg filter graph to process frames through this tile filter.

1. `bad.c:222` `filter_frame`
   - Entry point where `tile->out_ref` is allocated via `ff_get_video_buffer()` at line 230 when `tile->out_ref` is NULL

**Note:** The function is registered as a callback in the AVFilterPad struct (line 108: `int (*filter_frame)(struct AVFilterLink *link, struct AVFrame *frame)`) and gets invoked by FFmpeg's filter graph infrastructure when frames are processed. The cleanup path is `bad.c:275` `uninit`, which fails tofree `tile->out_ref`.
